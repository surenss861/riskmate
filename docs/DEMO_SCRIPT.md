# RiskMate Executive Demo Script (10 Minutes)

**Target Audience:** Insurers, Inspectors, Enterprise Buyers, Board Members  
**Goal:** Demonstrate enterprise-grade compliance platform without narration  
**Tone:** Confident, deliberate, professional

---

## Pre-Demo Setup (2 minutes)

### Environment Check
- [ ] All test data loaded (1-2 sample jobs)
- [ ] Templates created (hazard + job templates)
- [ ] Team members added
- [ ] Evidence uploaded (photos, documents)
- [ ] Permit pack generated (if Business plan)
- [ ] Browser zoom at 100%
- [ ] Full screen mode
- [ ] Network connection stable

### Opening State
- [ ] Dashboard page loaded
- [ ] No error messages visible
- [ ] All sections rendering correctly

---

## Demo Flow (8 minutes)

### 1. Create Job (1.5 minutes)

**What to Show:**
- Navigate to Jobs → Create Job
- Fill in client details (name, type, location)
- Select a job template
- Show template pre-filling hazards automatically
- Create job

**What to Say:**
> "RiskMate starts with jobs. Every job gets its own safety assessment. Templates let us spin up compliant jobs in seconds."

**Key Points:**
- Template system saves time
- Automatic hazard identification
- Risk score calculated immediately

**Success Indicators:**
- Job appears in list
- Risk score visible
- Status = Active

---

### 2. Apply Template (1 minute)

**What to Show:**
- Open job detail page
- Click "Apply Template"
- Select a hazard template
- Show hazards updating instantly
- Risk score recalculates

**What to Say:**
> "Templates standardize safety across jobs. When we apply a template, hazards and required controls update automatically. Everything is logged."

**Key Points:**
- Templates ensure consistency
- Changes are audited
- Risk score updates in real-time

**Success Indicators:**
- Hazards list updates
- Risk score changes
- Version history shows template application

---

### 3. Assign Worker (1 minute)

**What to Show:**
- Scroll to Job Assignment section
- Click "Assign Worker"
- Select worker from dropdown
- Show worker appearing in assigned list
- Hover over avatar to show tooltip

**What to Say:**
> "Job assignment creates accountability. Every worker assignment is logged for compliance. Managers can see who's responsible for each job."

**Key Points:**
- Clear accountability
- Audit trail for assignments
- Visual worker identification

**Success Indicators:**
- Worker appears in list
- Avatar shows correctly
- Assignment date visible

---

### 4. Approve Evidence (1.5 minutes)

**What to Show:**
- Navigate to Evidence Verification section
- Show pending evidence items
- Click "Approve" on a photo
- Show status changing to "Approved"
- Click "Reject" on another item
- Enter rejection reason
- Show rejection confirmation modal
- Show status changing to "Rejected"

**What to Say:**
> "Evidence verification is manager-only. Photos and documents must be approved before they're included in reports. Rejections require a reason for compliance."

**Key Points:**
- Manager oversight
- Required rejection reasons
- Status clearly visible
- All actions logged

**Success Indicators:**
- Approval updates instantly
- Rejection modal appears
- Status badges update
- Version history records actions

---

### 5. Generate Permit Pack (1.5 minutes)

**What to Show:**
- Navigate to Permit Packs section
- Click "Generate Permit Pack"
- Show progress modal (if Business plan)
- Show download starting automatically
- Show permit pack in history list
- Click "Download" on previous pack

**What to Say:**
> "Permit packs bundle everything an inspector needs: risk assessment, hazard checklist, mitigations, evidence, and metadata. One ZIP file, audit-ready."

**Key Points:**
- Complete compliance bundle
- Automatic ZIP generation
- Downloadable for offline use
- Version history

**Success Indicators:**
- Progress modal shows steps
- Download starts automatically
- Pack appears in history
- Download URL works

---

### 6. View Version History (1.5 minutes)

**What to Show:**
- Scroll to Version History section
- Show timeline with date grouping
- Click on a template reference
- Show evidence reference
- Show worker assignment reference
- Scroll through full history

**What to Say:**
> "Version history is the audit trail. Every change is logged: who did what, when, and why. This is what inspectors and insurers need to see."

**Key Points:**
- Complete audit trail
- Date grouping (Today, Yesterday, etc.)
- Clickable references
- Read-only for integrity

**Success Indicators:**
- Timeline renders correctly
- Dates grouped properly
- References are clickable
- All events visible

---

## Backup "Live Demo" Checklist

If screen recording fails or you need to demo live:

### Pre-Flight
- [ ] Clear browser cache
- [ ] Test all features work
- [ ] Have backup job data ready
- [ ] Know keyboard shortcuts
- [ ] Have error recovery plan

### During Demo
- [ ] Speak slowly and clearly
- [ ] Pause after each action
- [ ] Let animations complete
- [ ] Don't apologize for loading
- [ ] If something breaks, move on

### Recovery Scripts
- **If feature doesn't load:** "Let me show you another example..."
- **If error appears:** "That's logged for our team. Let's continue..."
- **If slow network:** "The data is loading. This is normal on slower connections..."

## Common Questions & Answers

### "What happens if something goes wrong?"

**Answer:**
> "Every action is logged in version history. If something goes wrong, we can see exactly what happened, when, and who was involved. All changes are reversible, and we maintain a complete audit trail."

**Key Points:**
- Version history shows everything
- Changes are reversible
- Complete audit trail
- No data loss

---

### "Is this audit-proof?"

**Answer:**
> "Yes. RiskMate is built for audits. Every job change is timestamped and logged. Evidence verification requires manager approval with reasons. Permit packs include complete metadata. Version history is read-only and cannot be edited. This is what inspectors and insurers need."

**Key Points:**
- Timestamped logs
- Manager approvals required
- Read-only audit trail
- Complete metadata
- Inspector-ready

---

### "Who touched what and when?"

**Answer:**
> "Version history shows exactly that. Every action includes: who performed it, when it happened, what changed, and why. For evidence verification, we also track who approved or rejected it and their reason. This creates a complete accountability trail."

**Key Points:**
- Version history shows all actions
- User attribution on every change
- Timestamps for everything
- Reasons for rejections
- Complete accountability
